Overview

Welcome to the deep-dive architectural guide for the Factory Agent. This chapter is designed for senior and principal engineers who need to understand the underlying mechanics, design philosophy, and strategic direction of the system. We move past "how it works" to "why it was built this way."

The Core Insight

The Factory Agent is not a chatbot; it is a Deterministic Stateful Graph.

The fundamental architectural insight is that every agent action must be a pure, traceable transformation of state. We treat the LLM as a "noisy CPU" and wrap it in a rigorous control flow that prevents side effects from occurring outside the Two-Phase Commit (2PC) protocol.

Logic Comparison (C# / Java vs. Python)

If you are coming from an Enterprise background, think of this as an Event-Sourced Saga Pattern with an LLM-based coordinator. Instead of hard-coded state transitions, we use probabilistic reasoning constrained by deterministic guardrails.

Architectural Tradeoffs

Choice Tradeoff Rationale
Graph over Chain Complexity vs. Reliability Linear chains fail on edge cases; graphs allow for repair loops and complex state transitions.
2PC over Immediate Latency vs. Safety Manufacturing side-effects are too expensive to "fix later." Staging is non-negotiable.
Relational Checkpointing Throughput vs. Durability MySQL ensures that even if a worker crashes mid-thought, the "ledger" of the session is never lost.

Strategic Direction

The ultimate goal is to transition from Human-in-the-Loop to Human-on-the-Loop. The architecture is designed to support 100% autonomous operations for routine tasks, while maintaining a "Manual Override" for anomalies detected by the Decision Guard.
