06. Guardrails & Safety

Zero-Trust Programmatic Validation

On this page

Overview

The Factory Agent implements a Zero-Trust policy between AI reasoning and physical execution. Every action proposed by the Planner must pass through a non-LLM, programmatic validator called the Decision Guard. This ensures that even if an LLM hallucinates, the system remains bound by hard-coded safety rules.

The Decision Guard Node

The decision_guard_node acts as a gatekeeper. It inspects the PlannerDecision and compares it against active user constraints and system-wide safety policies.

Programmatic Constraints

Constraints are explicit rules provided by the user or defined in the session context (e.g., "Do not modify machines in Section B").

  • Hard Filtering: The Guard checks every tool argument against the constraint list. If a match is found (e.g., machine ID is in Section B), the action is blocked.
  • Feedback Loop: When a constraint is violated, the Guard returns a specific error message to the Planner, explaining why the action was blocked so the Planner can find an alternative path.

Reference Validation

When the Planner uses symbolic references ($ref), the Guard ensures they are logically sound before they reach the execution pipeline.

Validation Description
Existence Ensures the referenced output actually exists in the history or the current bundle.
Circular Dependency Prevents Tool A from depending on the output of Tool A.
Type Consistency Verifies that the referenced output value matches the expected type for the current argument.

Risk Level Enforcement

The Guard evaluates the risk_level provided by the Planner but also performs its own independent assessment.

Escalation: If the Guard detects a write operation that the Planner labeled as "Low Risk," it automatically escalates it to write_commit, which may trigger a Human-in-the-Loop interrupt.
