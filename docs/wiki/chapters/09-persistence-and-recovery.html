09. Persistence & Recovery

Industrial State Survivability & Self-Healing Loops

On this page

Overview

Industrial AI requires 100% state survivability. If a server room loses power or a worker thread crashes, the Factory Agent must be able to resume its task exactly where it left off, without duplicate execution or data loss. This is achieved through a combination of atomic checkpointing and active session reconciliation.

Multi-Layered Persistence

Every node transition in the graph triggers a state snapshot. We use a hybrid storage model to balance performance and reliability.

Storage Layer Type Responsibility
MySQL Primary Permanent storage of the AgentState snapshots (pickled binary blobs).
Redis Cache Fast event bus for real-time SSE streaming and transient session metadata.

Self-Healing Recovery Sweep

The system is proactive about its own health. A dedicated background process handles sessions that have gone "stale" due to unexpected failures.

The Feature: A cold-start reconciliation logic that runs on server startup.

The Value: If the server crashes mid-execution, this logic automatically detects "Stuck" sessions and either attempts to resume them or moves them to a manual reconciliation queue, preventing industrial state leakage.

Ensuring Consistency

During recovery, the system doesn't just "blindly" restart. It performs a **Reconciliation Check**:

  • Backend Sync: Consults the Go Backend to see if any staged 2PC bundles were partially applied.
  • Idempotency Check: Re-calculates idempotency keys for the next pending step to ensure no duplicate API calls are made.
  • Truncation: If the state is corrupted, it can truncate to the last "Known-Good" checkpoint before the failure occurred.
