03. Intent Orchestration

Recursive Decomposition & Task Dependency Management

On this page

Overview

The Factory Agent handles complex, multi-part manufacturing requests by breaking them down into atomic work units called Intents. This "Divide and Conquer" approach ensures that even if one part of a request fails, the entire session remains consistent and manageable.

Intent Decomposition (Phase 3)

The Feature: A recursive intent_splitter that breaks compound queries into a dependency-aware stack.

The Value: Prevents agent failure on complex requests like "Find all high-priority jobs, delete them, then alert the supervisor." It ensures each step is planned and validated in the context of previous results.

Deconstruction Process

When a request enters the graph, the intent_splitter_node uses a specialized prompt to analyze the query's "topography".

Phase Action Outcome
1. Extraction Identify verbs and objects. Raw list of potential actions.
2. Dependency Search Look for pronouns or logic links (e.g. "them", "then"). A directed dependency graph of intents.
3. Constraint Injection Bind global rules to specific intents. Final Intent objects with local guardrails.

Dependency Mapping

graph LR Q[Query] --> S[Splitter] S --> I1[Intent 1: GET Machines] S --> I2[Intent 2: POST Jobs] I1 -.->|depends on| I2 I2 --> C[Cascade Controller]

Failure Cascading

To ensure industrial safety, the system implements a Cascade Cancellation mechanism. If an intent that provides critical data fails, all downstream intents that depend on that data are automatically aborted with a dependency_failure status.

  • Safe State: Prevents the agent from attempting to "delete all jobs" when the "get jobs" call failed to return a list.
  • Clear Context: The operator sees exactly why Step 2 didn't run, linked to the failure of Step 1.
