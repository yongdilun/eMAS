05. Two-Phase Commit (2PC)

Deterministic Transaction Management for Industrial Safety

On this page

Overview

The Two-Phase Commit (2PC) protocol is the primary safety mechanism of the Factory Agent. It ensures that no manufacturing side-effects occur without a validated dry-run and explicit commitment. This architecture exists to prevent partial failures where an agent might start a machine but fail to assign a job, leaving the hardware in an inconsistent state.

Execution Lifecycle

Phase 1: Staging (The Proposal)

When the planner_node proposes a write operation, the tool_pipeline does not execute it. Instead, it serializes the call and stashes it into the state.

  • Responsibility: Capture intent without side-effects.
  • Node: make_tool_execution_node in tool_pipeline.py.
Phase 2: Validation (The Dry-Run)

The entire staged_writes bundle is sent to the Go backend's dry-run endpoint. This simulates the transaction to catch logic errors or conflicts.

  • Responsibility: Ensure the backend is ready to accept the change.
  • Node: bundle_dry_run_node in planner_loop.py.
Phase 3: Commitment (The Execution)

Only after the dry-run succeeds (and optional human approval is granted), the commit_node sends the final execution signal.

  • Responsibility: Atomic application of the staged bundle.
  • Node: make_commit_node in tool_pipeline.py.

State Evolution

The AgentState acts as the persistent ledger for the transaction.

Field State at Staging State at Commitment
staged_writes Populated with PendingToolCall Cleared after success
tool_outputs Empty for write tools Appended with final result
status awaiting_dry_run completed or idle

Design Tradeoffs

  • Pro: Atomicity. Prevents the "partial success" nightmare common in LLM agents.
  • Pro: Idempotency. The 2PC bundle carries a unique session-based key.
  • Con: Latency. Requires an extra round-trip to the backend before commitment.
